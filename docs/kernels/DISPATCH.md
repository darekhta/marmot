# Kernel Dispatch Flow

This document describes how Marmot routes operations to kernel implementations via the unified bytecode path.

---

## 1. Overview

Marmot has a single runtime dispatch mechanism: **bytecode execution**. All operations -- whether invoked through the graph executor or the public C API -- go through the same backend-specific bytecode interpreter. There are no per-op switch statements or separate dispatch paths.

The dispatch system has three layers:

1. **Signature construction** -- Build a `marmot_op_signature_t` describing the operation.
2. **Kernel selection** -- Query the backend to find a matching kernel and get a dense `op_index`.
3. **Bytecode execution** -- Use the `op_index` as an array index into a function pointer table.

Both the graph path and the C API path use the same kernel query functions, the same bytecode tables, and the same exec functions. They differ only in how the bytecode program is constructed.

---

## 2. C API Path (Ad-hoc Dispatch)

When the public C API executes a single operation (e.g., `marmot_matmul`, `marmot_add`):

```
  C API call (e.g. marmot_add)
       |
       v
  Build marmot_op_signature_t + pack args
       |
       v
  marmot_bc_try_execute_signature()           [src/core/dispatch/dispatch_execute.c]
       |
       +-- marmot_bc_compile_signature()      [src/core/bytecode/bytecode_compile.c]
       |       |
       |       +-- marmot_backend_query_kernel_with_fallback()
       |       |       |                      [src/core/dispatch/kernel_query.c]
       |       |       +-- marmot_cpu_query_kernel()   (generated)
       |       |       +-- marmot_metal_query_kernel() (generated)
       |       |
       |       +-- Returns marmot_bc_selection_t { op_index, resolved_sig }
       |
       +-- marmot_bc_execute_op(backend, op_index, exec_ctx, args)
               |
               +-- exec_table[op_index](exec_ctx, args)
```

The compile step is cached: repeated calls with the same signature reuse the previously resolved `op_index` without re-querying the backend.

---

## 3. Graph Path (Bytecode Program)

When executing a finalized graph, the flow separates into two phases:

### Phase A: Graph Finalization (compile time)

```
  Graph::finalize_impl()                       [src/graph/graph.cpp]
       |
       +-- For each node:
       |       |
       |       v
       |   Build marmot_op_signature_t from node metadata
       |       |
       |       v
       |   query_backend_for_node()            [src/graph/kernel_query.cpp]
       |       |
       |       +-- marmot_backend_query_kernel(backend, &sig, &caps)
       |       |                               [src/core/dispatch/kernel_query.c]
       |       |
       |       +-- Returns marmot_kernel_selection_t { kernel_id, op_index }
       |       |
       |       v
       |   Store node.kernel_id, node.bc_op_index
       |       |
       |       v
       |   Encode op_index + immediates into marmot_bc_program_t
       |
       +-- Apply fusion passes (may merge nodes, adjust opcodes)
       |
       +-- Finalize bytecode program
```

### Phase B: Graph Execution (runtime)

```
  graph_executor.execute()                     [src/graph/graph_executor.cpp]
       |
       v
  marmot_bc_execute(&program, &exec_ctx, regs) [src/core/bytecode/bytecode.c]
       |
       +-- For each opcode in program:
               |
               v
           Read op_index from bytecode stream
               |
               v
           exec_table[op_index](exec_ctx, packed_immediates)
```

## 4. The Bytecode Interpreter

The interpreter is implemented in `src/core/bytecode/bytecode.c`. It reads a linear stream of bytecode instructions, where each instruction is:

```
+----------+-----------+
| op_index | immediates|
| (uint16) | (N bytes) |
+----------+-----------+
```

The immediate size for each `op_index` is known from the `imm_size` table (generated). The interpreter advances through the program by reading `op_index`, looking up the immediate size, extracting the packed arguments, and calling:

```c
exec_table[op_index](exec_ctx, &immediates)
```

### Per-Backend Tables

Each backend provides a `marmot_bc_tables_t` structure (via `marmot_bc_get_tables()`):

```c
typedef struct marmot_bc_tables {
    const uint16_t *imm_size;          // imm_size[op_index] = bytes of immediates
    const marmot_bc_exec_fn *exec_table; // exec_table[op_index] = function pointer
    const marmot_bc_schema_id_t *schema_id; // schema_id[op_index] = arg schema
    uint16_t op_count;                 // total number of dense opcodes
} marmot_bc_tables_t;
```

The `exec_table` entries are generated functions that unpack the packed immediates into typed arguments and call the actual kernel implementation. These are generated by the `bytecode_exec.c.j2` template.

### Dense Opcodes

Because `op_index` values are dense (0, 1, 2, ..., N-1), the interpreter uses plain array indexing with no hash tables, binary searches, or switch statements. This gives O(1) dispatch overhead per instruction.

## 5. Backend Dispatch: CPU vs Metal

### CPU Bytecode Interpreter

- Source: `src/backends/cpu/dispatch/bytecode_exec_cpu.gen.c`
- Tables: `src/backends/cpu/dispatch/bytecode_tables_cpu.gen.h` + `.gen.c`
- Exec functions call into CPU kernel implementations (scalar, NEON, AVX2, or Accelerate depending on the selected profile).
- Platform guards (`#if MARMOT_ENABLE_NEON`, etc.) compile out unavailable profiles.

### Metal Bytecode Interpreter

- Source: `src/backends/metal/ops/bytecode_exec_metal.gen.mm` (~1.25MB)
- Tables: `src/backends/metal/ops/bytecode_tables_metal.gen.h` + `.gen.mm`
- Exec functions encode Metal compute commands: set pipeline state, bind buffers, dispatch threadgroups.
- The Metal exec functions are Objective-C++ and interact with the Metal runtime directly.

Both backends share the same interpreter loop in `src/core/bytecode/bytecode.c`. The backend-specific tables are loaded once via `marmot_bc_get_tables()`.

## 6. Fallback Behavior

### Variant Flags Fallback

Kernel query supports a limited fallback rule:
- If `sig->variant_flags != 0` and the backend does not support those requested variants, query retries with `variant_flags = 0`.

This is implemented in `marmot_backend_query_kernel_with_fallback()` (`src/core/dispatch/kernel_query.c`).

### "No Kernel" Handling

Different layers signal "no matching kernel" differently:

- **Kernel query** returns `marmot_kernel_selection_t{ .supported = false, .fallback_reason = "..." }`.
- **Graph finalization** converts "unsupported" into `MARMOT_ERROR_NOT_IMPLEMENTED` and sets a detailed error message (`marmot_get_last_error_detail()`).
- **Ad-hoc bytecode execution** returns an error when compilation or execution fails (often `MARMOT_ERROR_NOT_IMPLEMENTED`).

### Stride Mode / Layout Legalization

Graph finalization performs layout legalization:
- If a node has no supported kernel for the given tensor layouts, it may insert contiguous copies or relax stride constraints, then retry selection.

This logic lives in `Graph::finalize_impl()` (`src/graph/graph.cpp`).

## 7. Codegen File Map

### CPU

| File | Role |
|------|------|
| `src/backends/cpu/dispatch/cpu_kernel_query.gen.c` | Signature -> kernel selection |
| `src/backends/cpu/dispatch/bytecode_tables_cpu.gen.h` | Opcode counts, immediate sizes |
| `src/backends/cpu/dispatch/bytecode_tables_cpu.gen.c` | Dense opcode table data |
| `src/backends/cpu/dispatch/bytecode_exec_cpu.gen.c` | op_index -> exec function table |

### Metal

| File | Role |
|------|------|
| `src/backends/metal/internal/metal_kernel_query.gen.mm` | Signature -> kernel selection |
| `src/backends/metal/ops/bytecode_tables_metal.gen.h` | Opcode counts, immediate sizes |
| `src/backends/metal/ops/bytecode_tables_metal.gen.mm` | Dense opcode table data |
| `src/backends/metal/ops/bytecode_exec_metal.gen.mm` | op_index -> exec function table |

### Shared

| File | Role |
|------|------|
| `src/core/dispatch/kernel_query.c` | Backend query router + fallback logic |
| `src/core/bytecode/bytecode.c` | Bytecode interpreter loop |
| `src/core/bytecode/bytecode_compile.c` | Ad-hoc compile + cache |
| `src/core/dispatch/dispatch_execute.c` | C API entry point for ad-hoc dispatch |
